<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Orchestrator → Worker (registration response) | Orchestrator provides worktree_path to worker; path must point to valid worktree created in Plan 01 |
| Worker → Filesystem (worktree) | Worker receives worktree_path and uses it for all filesystem operations; path must be within workspace to prevent directory traversal |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Information Disclosure | worktree_path in registration response | accept | Registration happens over localhost HTTP; no TLS needed for v1; paths are not secrets (workers need to know their own directories) |
| T-02-06 | Tampering | Worker modifying worktree_path after receiving | accept | Workers are trusted processes spawned by orchestrator; no sandboxing at process level for v1 |
| T-02-07 | Spoofing | Malicious worker registering with fake side | mitigate | Registration handler validates worker identity against expected dyad config; only expected workers accepted (existing logic) |

**No new threats introduced by worktree_path field.** Threat surface unchanged from existing workspace field; both are paths provided by trusted orchestrator to trusted workers over localhost.
</threat_model>
