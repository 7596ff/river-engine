**Function 2: ensure_worktree_exists**

Signature:
```rust
async fn ensure_worktree_exists(
    repo_path: &Path,
    worktree_name: &str,
    branch_name: &str,
) -> Result<(), SupervisorError>
```

Implementation (per D-06 reuse logic):

1. Construct worktree_path = repo_path.join(worktree_name)
2. If worktree_path exists AND is_valid_worktree returns true:
   - Log: "Reusing existing worktree {worktree_name} at {worktree_path}"
   - Return Ok(())
3. Else if worktree_path exists but is_valid_worktree returns false:
   - Log warning: "Invalid worktree at {worktree_path}, cleaning"
   - Call tokio::fs::remove_dir_all(&worktree_path).await
   - Continue to step 4
4. Create branch if it doesn't exist (ignore error if branch exists):
   ```rust
   Command::new("git")
       .args(&["-C", repo_str, "branch", branch_name])
       .stdout(Stdio::null())
       .stderr(Stdio::null())
       .output()
       .await
       .ok(); // Ignore error - branch may already exist
   ```
5. Create worktree:
   ```rust
   let output = Command::new("git")
       .args(&["-C", repo_str, "worktree", "add", worktree_str, branch_name])
       .output()
       .await?;
   ```
   Note: Use `worktree add <path> <branch>` NOT `worktree add -b <branch> <path>` when branch exists (avoids "already exists" error).
6. Check output.status.success():
   - If false: return Err(SupervisorError::WorktreeCreationFailed(String::from_utf8_lossy(&output.stderr).into()))
7. Log: "Created worktree {worktree_name} at {worktree_path} on branch {branch_name}"
8. Return Ok(())

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Orchestrator → Git CLI | Orchestrator spawns git processes with workspace paths from config; paths must be validated to prevent directory traversal |
| Filesystem (worktrees) → Workers | Workers read/write to worktree directories; isolation enforced by git branch separation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | ensure_worktree_exists (path validation) | mitigate | Validate workspace path is absolute or resolve to absolute before passing to git; reject paths containing ".." or symlinks via `canonicalize()` |
| T-02-02 | Denial of Service | Git command failures (worktree add) | accept | Git command errors propagate to spawn_dyad; orchestrator logs error and dyad fails to start; no silent degradation |
| T-02-03 | Information Disclosure | Worktree contents readable by same-user processes | accept | Deployment is localhost-only, single-user; filesystem permissions controlled by deployment environment (NixOS) |
| T-02-04 | Tampering | Stale worktree metadata causing data loss | mitigate | is_valid_worktree checks .git file validity; invalid worktrees cleaned before recreation; manual `git worktree repair` available if needed |

**Threat mitigations implemented in code:**
- T-02-01: Workspace path comes from DyadConfig loaded from orchestrator config; no user input. Path validation could be added in future if config becomes dynamic.
- T-02-04: is_valid_worktree() validates .git file contains "gitdir:" to detect linked worktrees vs stale directories.
</threat_model>
