**Structure and content (per D-01, D-02 from CONTEXT.md):**

1. **Introduction section:**
   - Explain PR-style flow: agents work on feature branches (left/right), merge to main to share
   - Reference Phase 2 infrastructure: worktrees at workspace/left/ and workspace/right/
   - Link to `reference.md` for bash tool syntax

2. **## When to Commit (per D-04, D-05, D-06, D-07):**
   - Commit frequently on your own branch (left or right) after substantive changes
   - Agent judges what counts as "substantive" — guideline: "commit after meaningful writes or before transitions"
   - Each commit on feature branch is a working snapshot, not a release
   - Use squash merge when merging to main (keeps shared history clean)
   - Commit message content: agent decides based on what changed (no strict format, but examples provided)
   - Example commands:
     ```bash
     git status  # See what changed
     git add workspace/notes/working-notes.md workspace/embeddings/topic-x.md
     git commit -m "notes: captured user feedback on architecture, three new insights"
     ```

3. **## When to Sync (per D-08, D-09, D-10, D-11):**
   - Default: sync at turn start (before acting or after spectating)
   - Mandatory: sync before responding to external messages (user or Ground)
   - Optional: additional syncs at agent discretion when fresh state matters
   - "PR-style" workflow using local git commands:
     - Agent merges feature branch to main (conceptually "opening a PR")
     - Partner pulls main to see changes (conceptually "reviewing the PR")
   - Example workflow:
     ```bash
     # At turn start: pull partner's latest changes
     git pull origin main

     # After your work: merge to main to share
     git checkout main
     git merge --squash left -m "actor: processed inbox, captured patterns"

     # Partner syncs by pulling
     git pull origin main
     ```

4. **## File Ownership Convention (per D-15, D-16):**
   - Purpose: Minimize conflicts by establishing ownership patterns
   - Actor owns: `notes/`, `artifacts/`, conversation writes
   - Spectator owns: `moves/`, `moments/`, `embeddings/` (curation)
   - Both may write to embeddings (actor captures, spectator curates) — usually sequential
   - Ownership by convention, not enforcement — agents respect boundaries but can cross when needed
   - Table format:
     | Directory | Owner | Purpose |
     |-----------|-------|---------|
     | notes/ | Actor | Working notes, drafts, scratch space |
     | artifacts/ | Actor | Generated files, code, documents |
     | conversations/ | Actor | Chat history writes (both read) |
     | moves/ | Spectator | Per-turn summaries |
     | moments/ | Spectator | Arc summaries |
     | embeddings/ | Both | Actor captures, spectator curates |

5. **## Conflict Resolution (per D-12, D-13, D-14):**
   - Agent attempts autonomous resolution first
   - Inspect conflict markers (<<<<<<< HEAD, =======, >>>>>>>)
   - Read both sides, understand intent from each agent's role context
   - Resolve by: keeping both (if complementary), choosing one (if clear authority), or synthesizing
   - Escalate to Ground if: cannot determine correct version, conflict reflects fundamental disagreement, or prevents progress
   - Escalation format (per D-13, Claude's discretion):
     - Notify Ground via backchannel
     - Include: file path, conflict markers (from `git diff`), what each side was trying to do, why can't resolve autonomously, what you need from Ground
   - Example commands:
     ```bash
     # Pull encounters conflict
     git pull origin main
     # Output: CONFLICT (content): Merge conflict in workspace/embeddings/topic-x.md

     # Inspect conflict
     git status  # Shows unmerged paths
     git diff workspace/embeddings/topic-x.md  # Shows conflict markers

     # After resolving (editing file manually):
     git add workspace/embeddings/topic-x.md
     git commit -m "resolve: topic-x, merged actor notes with spectator curation"
     ```

6. **## Common Operations:**
   - Start session with sync
   - Commit substantive changes
   - Merge to main (share with partner)
   - Pull partner's changes
   - Detect and resolve conflicts
   - Each operation: bash commands with explanations
   - Use examples from RESEARCH.md sections "Common Operation 1" through "Common Operation 5"

7. **## Anti-Patterns:**
   - Merge conflicts without reviewing (always inspect both sides)
   - Vague commit messages ("wip", "updates" — prefer specific descriptions)
   - Forgetting to pull before merging (always sync first)
   - Force pushing to main (never use --force, main is shared history)
   - Leaving conflicts unresolved (blocks subsequent operations)

**Documentation style:**
- Follow `reference.md` pattern: tables for structured info, code blocks for commands, clear headers
- Concrete examples with actual bash commands (not pseudocode)
- Explain the "why" alongside the "how" (e.g., "squash merge keeps shared history clean")
- Agent perspective ("you commit", "your partner pulls", not "the system does")

**Claude's discretion areas (from CONTEXT.md):**
- Exact git flags (e.g., `--squash`, `-m`, `--no-ff`) — use standard, safe options
- Additional sync scenarios beyond core cases — include edge cases if helpful (e.g., interrupted merge, recovering from mistakes)
- Conflict escalation artifact format — structured enough to be useful, flexible enough for context

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent→git | Agents execute git commands via bash tool; commands could be malformed or malicious |
| worktree→filesystem | Git operations modify shared workspace; symlink attacks possible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Git command injection via bash tool | accept | Bash tool already uses Command::new("sh") with .arg() (safe); sync.md instructs literal commands, not string interpolation |
| T-03-02 | Information Disclosure | Conflict markers expose both agents' work | accept | Agents are trusted; Ground has full access anyway; conflict content is operational data |
| T-03-03 | Denial of Service | Agent creates invalid merge, blocks partner | accept | Git merge is transactional (succeeds fully or fails); partner can abort merge; Ground can intervene |
| T-03-04 | Tampering | Symlink attack via git checkout | mitigate | Phase 2 created worktrees with validated paths; sync.md will not instruct symlink operations; git respects worktree boundaries |
| T-03-05 | Elevation of Privilege | Agent modifies partner's owned files | accept | Ownership by convention, not enforcement; Ground supervises; agents trusted processes |
</threat_model>
