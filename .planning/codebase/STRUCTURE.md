# Codebase Structure

**Analysis Date:** 2026-04-06

## Directory Layout

```
river-engine/
├── crates/                          # Cargo workspace members
│   ├── river-orchestrator/          # Process supervisor and registry
│   ├── river-worker/                # Agent runtime (think→act loop)
│   ├── river-protocol/              # Foundational types (zero inter-crate deps)
│   ├── river-adapter/               # Worker ↔ adapter type interface
│   ├── river-context/               # Context assembly (pure function)
│   ├── river-snowflake/             # Distributed ID generation
│   ├── river-discord/               # Discord adapter (example)
│   ├── river-embed/                 # Embedding/search service
│   └── river-tui/                   # Terminal UI for observing state
├── archive/                         # Deprecated code (do not modify)
├── docs/                            # Design documents and specs
├── .planning/                       # GSD planning documents (this directory)
├── shell.nix                        # Nix development environment
├── Cargo.toml                       # Workspace manifest
└── Cargo.lock                       # Dependency lock
```

## Directory Purposes

**`crates/river-orchestrator/`:**
- Purpose: Process spawning, supervision, registration, health checks
- Contains: Process supervisor, HTTP router, registry, respawn manager, config loader
- Key files: `main.rs` (entry), `supervisor.rs`, `registry.rs`, `http.rs`, `config.rs`, `respawn.rs`

**`crates/river-worker/`:**
- Purpose: Agent runtime for the think→act loop
- Contains: Worker loop, LLM client, tool execution, workspace loading, conversation persistence
- Key files: `main.rs` (entry), `worker_loop.rs`, `llm.rs`, `tools.rs`, `state.rs`, `workspace_loader.rs`, `persistence.rs`, `http.rs`

**`crates/river-protocol/`:**
- Purpose: Foundational types used by all crates (no dependencies on other river-* crates)
- Contains: `Side`, `Baton`, `Channel`, `Ground`, `Author`, `ModelConfig`, registration/response types, conversation format
- Key files: `lib.rs`, `identity.rs`, `model.rs`, `registration.rs`, `registry.rs`, `conversation/mod.rs`

**`crates/river-adapter/`:**
- Purpose: Types-only interface for worker ↔ adapter communication
- Contains: Feature system (`FeatureId`, `OutboundRequest`), event types (`InboundEvent`), response types, OpenAPI schema
- Key files: `lib.rs`, `feature.rs`, `event.rs`, `response.rs`, `traits.rs`, `error.rs`, `openapi.json`

**`crates/river-context/`:**
- Purpose: Pure function to assemble workspace into LLM-compatible messages
- Contains: `build_context()`, message formatting, token estimation, workspace types
- Key files: `lib.rs`, `assembly.rs`, `format.rs`, `tokens.rs`, `request.rs`, `response.rs`, examples (test runners)

**`crates/river-snowflake/`:**
- Purpose: Distributed ID generation with embedded timestamps
- Contains: Snowflake generator, `AgentBirth` metadata encoding
- Key files: `lib.rs`, `main.rs` (server), `snowflake/` submodule

**`crates/river-discord/`:**
- Purpose: Discord adapter (example implementation of adapter pattern)
- Contains: Twilight gateway client, HTTP adapter interface, event→request translation
- Key files: `main.rs` (entry), `discord.rs`, `http.rs`

**`crates/river-embed/`:**
- Purpose: Vector embedding and semantic search service
- Contains: Chunking, embedding client, vector store (SQLite with sqlite-vec), search logic
- Key files: `main.rs` (entry), `store.rs`, `embed.rs`, `chunk.rs`, `search.rs`, `http.rs`

**`crates/river-tui/`:**
- Purpose: Terminal UI for observing worker/adapter state
- Contains: Ratatui-based UI, HTTP client for querying orchestrator/workers
- Key files: `main.rs` (entry), `tui.rs`, `http.rs`, `adapter.rs`

**`docs/`:**
- Purpose: Design documentation, architecture specs, analysis
- Key files: `WORKER-DESIGN.md` (worker loop philosophy), `GAP-ANALYSIS.md`, `archive/` for historical specs

**`archive/`:**
- Purpose: Deprecated code from earlier iterations (do not modify)
- Contains: Old single-binary versions of orchestrator, adapters, tools, etc.
- Note: Not built in current Cargo workspace

## Naming Conventions

**Files:**
- Rust source files: lowercase with underscores (`worker_loop.rs`, `llm.rs`)
- Cargo manifests: `Cargo.toml`
- Config examples: `river.json`, `example-config.nix`
- Tests: colocated with source or in `tests/` directory

**Directories:**
- Crates: lowercase with hyphens (`river-orchestrator`, `river-worker`)
- Modules: lowercase with underscores (reflect in `mod.rs` or file structure)
- Internal modules: `src/` contains modules, no extra nesting for single-module crates

**Types:**
- Structs/Enums: PascalCase (`WorkerState`, `ProcessKey`, `OutboundRequest`, `FeatureId`)
- Traits: PascalCase (`Adapter`)
- Functions: snake_case (`build_context`, `spawn_worker`, `run_loop`)
- Constants: UPPER_SNAKE_CASE (used sparingly; mostly in tests)
- Lifetimes: Single lowercase letters or descriptive (`'a`, `'reg`)

**Serialized Forms (JSON/YAML):**
- Field names: snake_case (via `#[serde(rename_all = "snake_case")]`)
- Enum variants: snake_case (e.g., `"send_message"` serializes from `SendMessage`)
- Examples: `OutboundRequest::SendMessage` → `{"send_message": {...}}`

## Where to Add New Code

**New Feature:**
- Primary code: `crates/river-worker/src/` (if worker logic) or `crates/river-*/src/` (if in specific crate)
- Tests: `crates/river-*/tests/` for integration or `#[cfg(test)]` module in source file

**New Component/Module:**
- Implementation: Depends on scope:
  - Shared types: `crates/river-protocol/src/` (no inter-crate dependencies)
  - Adapter interface: `crates/river-adapter/src/`
  - Worker runtime: `crates/river-worker/src/`
  - Orchestrator logic: `crates/river-orchestrator/src/`
  - New adapter: Create `crates/river-{adapter-name}/` with Cargo.toml inheriting from workspace
- Cargo.toml: Declare dependency via `{ path = "../{sibling-crate}" }` (workspace dependencies)

**Utilities:**
- Shared helpers: `crates/river-protocol/src/` if used across crates, or internal module in crate that owns the logic
- Example: `extract_timestamp()` in `crates/river-context/src/id.rs` is used by worker, so could be in protocol, but currently in context (acceptable since workers import context anyway)

**HTTP Endpoints:**
- Orchestrator endpoints: `crates/river-orchestrator/src/http.rs` → add route in `router()` function
- Worker endpoints: `crates/river-worker/src/http.rs` → add route in `router()` function
- Adapter endpoints: `crates/river-{adapter}/src/http.rs` → add route (example: `crates/river-discord/src/http.rs`)

## Special Directories

**`docs/`:**
- Purpose: Design specifications, analysis, rationale
- Generated: No (human-written)
- Committed: Yes
- Contains: `WORKER-DESIGN.md`, `GAP-ANALYSIS.md`, `archive/` (historical specs)

**`.planning/codebase/`:**
- Purpose: GSD planning documents (generated by `/gsd-map-codebase`)
- Generated: Yes (by mapper agents)
- Committed: Yes (for continuity across sessions)
- Contains: `ARCHITECTURE.md`, `STRUCTURE.md`, `CONVENTIONS.md`, `TESTING.md`, `CONCERNS.md`, `STACK.md`, `INTEGRATIONS.md`

**`archive/`:**
- Purpose: Old versions of binaries from before monorepo migration
- Generated: No (historical code)
- Committed: Yes (for reference, not built)
- Note: Do not modify; kept for lineage tracing only

**`target/`:**
- Purpose: Cargo build artifacts
- Generated: Yes (by `cargo build`)
- Committed: No (in `.gitignore`)

**`crates/river-context/examples/`:**
- Purpose: Runnable examples for testing context assembly
- Generated: No (reference implementations)
- Committed: Yes
- Run via: `cargo run --example {name}` from crate root
