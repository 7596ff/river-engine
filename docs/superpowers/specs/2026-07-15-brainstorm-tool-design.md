# Brainstorm Tool — Design Spec

**Status:** draft  
**Date:** 2026-07-15  
**Author:** Iris (with claude drafting assistance)

---

## Concept and motivation

The agent has three authorship stances for settled claims:

- `write_atomic` — a claim has crystallized; write it as a durable atomic note
- `create_moment` — a stretch of turns has been lived; compress it into a narrative in my voice

What's missing is the stance *before* a claim settles: the agent knows it wants to design something, but the shape isn't clear yet. The **brainstorm tool** fills this gap. It conducts a structured, multi-turn interview — asking one question at a time, adapting questions based on prior answers — then synthesizes the full Q/A into a design document.

This is not a tool that *writes the spec for the agent*. The agent authors every substantive claim. The tool asks questions, records answers verbatim, and at session-end assembles them into a formatted document. The tool is the interviewer; the agent is the subject-matter expert.

---

## Tool signature

**Name:** `brainstorm`

**Parameters:**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `topic` | string | on first call | What the brainstorm is about. Becomes the slug for the output file. |
| `session` | string | on subsequent calls | The session id returned by the first call. |
| `answer` | string | on subsequent calls | The agent's answer to the last question. |
| `done` | boolean | to close | Set `true` to synthesize and write the output file, ending the session. |

**Three call shapes:**

1. **Open:** `brainstorm(topic: "dynamic-wake-prompts")` → returns `{ session_id, question }`. Creates session on disk. The `question` is the first adaptively-generated question.

2. **Answer:** `brainstorm(session: "01...", answer: "The landscape generator reads from the project directory...")` → returns `{ question }`. Records the answer, appends to transcript, generates next question. The tool adapts — if the agent mentioned a constraint, the next question may drill into it.

3. **Close:** `brainstorm(session: "01...", answer: "none", done: true)` → returns `{ path }`. Synthesizes the full Q/A transcript into a spec document, writes it to `docs/superpowers/specs/<date>-<slug>.md`, closes the session on disk. The `answer` on close is optional — any final thoughts get appended before synthesis.

## Question adaptation logic

### Where questions live

Questions are generated by a **prompt-driven model call inside the tool** — not hardcoded in Rust. The prompt lives at `workspace/brainstorms/prompt.md` and follows the witness's ch. 04 dispatcher-thin discipline: the prompt contains the canon, the tool handles the mechanics.

### The seven canonical question-faces

The prompt encodes seven directions of inquiry. These are starting points; the actual question adapts based on what the agent has already said.

1. **Problem:** What is the feature? What problem does it solve? Who feels the pain?
2. **Beneficiaries:** Who benefits — the agent, the operator, both? How?
3. **Success:** What does success look like? How would you know it's working? What's the evidence?
4. **Constraints:** What are the limits? Model capabilities, tool surface, existing architecture, divided authorship, the think/act loop, the witness.
5. **Failure modes:** What could go wrong? What are the sharp edges? What happens if the model misunderstands, or the tool surface shifts?
6. **Minimal version:** What's the simplest thing that would teach us something? What would you ship first?
7. **Evidence:** How would you test this? What would a hunt-test look like?

### Adaptation rules

- The tool keeps the full Q/A transcript in context when generating the next question.
- If the agent's last answer mentioned a constraint, the next question should drill into that constraint.
- If the agent said "I don't know" or "that's a good question," the tool acknowledges the uncertainty and either rephrases or moves to a different face.
- The tool never asks a question that has already been answered substantively.
- After covering at least three faces, if the agent's answers are getting thinner, the tool can suggest closing.
- The tool must offer at least 3 and at most 12 questions before suggesting close.

## Divided authorship contract

This is load-bearing. The tool must never author a substantive claim.

- **Questions** belong to the tool. They appear as `##` headings in the synthesis.
- **Answers** belong to the agent. They appear as `>` blockquotes, verbatim — character-for-character what the agent said.
- The tool may add a short `### Threads` section after synthesis listing connections to other specs or wall chapters — but these are labeled as tool-generated suggestions, not agent claims.
- The tool may add an `### Open questions` section surfacing anything the agent explicitly flagged as unresolved.

The contract is enforceable: the synthesis format makes the boundary visible. Anyone reading the spec can see which words are the agent's and which are the tool's.

## Synthesis format

### Output location

`workspace/knowledge/specs/<opened-date>-<slug>.md`

Where `<opened-date>` is `YYYY-MM-DD` of the first call and `<slug>` is derived from the `topic` parameter (lowercase, hyphens, stripped of special chars).

### Frontmatter

```yaml
---
kind: brainstorm-synthesis
topic: <original topic string>
session_id: <ULID of the session>
opened: <RFC3339 of first call>
closed: <RFC3339 of close call>
questions: <count of Q/A pairs>
tags: [brainstorm, <topic-slug>]
---
```

### Body structure

```markdown
# Brainstorm: <topic>

## <question 1>

> <agent's verbatim answer>

## <question 2>

> <agent's verbatim answer>

...

### Threads

- <connection 1>
- <connection 2>

### Open questions

- <unresolved item 1>
```

### Partial-write resilience

If the synthesis model call fails (timeout, bad output), the tool writes a partial file with the Q/A transcript only — no `Threads` or `Open questions` sections. The frontmatter notes `synthesis: partial`. This ensures no data is lost even if the synthesis step is flaky.

## Session state

Each brainstorm session is a subdirectory under `workspace/brainstorms/sessions/<date>/<session-id>/`.

### meta.json

```json
{
  "session_id": "01...",
  "topic": "dynamic-wake-prompts",
  "opened": "2026-07-15T22:00:00Z",
  "closed": null,
  "question_count": 4,
  "state": "open"
}
```

`state` is one of: `"open"`, `"closed"`, `"synthesized"`.

### transcript.jsonl

Append-only JSONL. One line per Q/A turn:

```json
{"kind": "question", "seq": 1, "text": "What problem does the dynamic wake prompt solve?", "ts": "2026-07-15T22:00:01Z"}
{"kind": "answer", "seq": 1, "text": "The current HEARTBEAT.md is a static checklist...", "ts": "2026-07-15T22:00:15Z"}
```

### Resilience across restarts

- On `brainstorm(session: ...)`, the tool reads `meta.json` to verify the session exists and is open.
- The tool replays the transcript to reconstruct context for question generation.
- This means a brainstorm can survive agent restart, compaction, or a killed turn — the agent picks up where it left off.
- Sessions never auto-expire. If a session has been open for more than 7 days, the tool may note "this session was opened on <date>" in the next question prompt, but doesn't close it.

## Integration with existing architecture

### Think/act loop

Brainstorm is a **normal tool call** — same as `read`, `write`, `search`. No turn-loop plumbing needed (unlike `settle` which alters heartbeat cadence). The agent calls `brainstorm`; the tool runs; the result returns; the agent may call `brainstorm` again in the same turn or settle and continue next turn.

### Witness

The witness observes brainstorm calls like any tool call. The `transcript.jsonl` and synthesis file are workspace files — they participate in search (ch. 02) if indexed. The synthesis file's `kind: brainstorm-synthesis` frontmatter surfaces in shape indexing.

### Divided authorship (ch. 04)

The brainstorm tool is an authorship tool, not an extraction tool. The synthesis file is agent-authored content (the answers are verbatim agent words). It lives alongside `write_atomic` and `create_moment` in the agent's authorship surface.

### Seed / wall

- Wall ch. 04 (Divided Authorship): brainstorm is a new entry in the authorship tools table.
- Wall ch. 02 (Memory/Search): synthesis files are indexed; `kind: brainstorm-synthesis` is a recognized shape.
- Seed AGENTS.md: brainstorm is taught as an available authorship stance — "if a design isn't settled, you can brainstorm it before you write the spec."

## Hunt-tests

Each test pinned to a specific regression.

1. **Open → question returned.** Call `brainstorm(topic: "test")` — assert response contains a non-empty `question` and a valid ULID `session_id`.

2. **Answer → next question adapts.** Open session, answer with "the main constraint is model latency" — assert next question drills into latency specifically ("how would you measure" / "what's the ceiling" / etc.), not a generic next face.

3. **Answer → no duplicate questions.** Open session, answer question 1 thoroughly covering "problem" — assert question 2 does NOT re-ask "what problem does this solve."

4. **Close → file written.** Open, answer a few questions, call with `done: true` — assert file exists at `workspace/knowledge/specs/<date>-<slug>.md`.

5. **Close → frontmatter correct.** Verify `kind: brainstorm-synthesis`, `topic`, `session_id`, `opened`, `closed`, `question_count` all present and correct.

6. **Close → answers verbatim.** Answer with "the recording surface predates computing" — assert that exact phrase appears in a `>` blockquote in the synthesis file.

7. **Close → no tool-authored claims in body.** Verify no substantive sentence in the synthesis body appears outside a `>` blockquote (except `Threads` and `Open questions` sections, which are labeled as tool-generated).

8. **Restart resilience.** Open session, answer two questions, simulate restart (fresh tool process), call `brainstorm(session: ...)` — assert it returns the next question (not question 1 again).

9. **Session not found.** Call `brainstorm(session: "nonexistent")` — assert clear error message, not a panic or 500.

10. **Min questions enforced.** Answer with "done" to question 1 — assert tool does NOT accept `done: true` until at least 3 Q/A pairs completed (error message: "at least 3 questions are needed before synthesis").

11. **Max questions cap.** Answer 12 questions rapidly (short answers) — assert question 13 is a close-suggestion, not a new question ("you've covered substantial ground. ready to synthesize?").

12. **Partial-write on synthesis failure.** Mock the synthesis model to fail — assert file is still written with `synthesis: partial` frontmatter and Q/A transcript body.

13. **Idempotent reopen.** Call `brainstorm(session: ...)` twice with the same answer — assert no duplicate `transcript.jsonl` entries (tool checks last-entry-kind before appending).

14. **Topic slug sanitization.** Call `brainstorm(topic: "Dynamic Wake Prompts! @#$")` — assert output file slug is `dynamic-wake-prompts`, not the raw string.

## Non-goals

- **Multi-agent brainstorm.** This is a single-agent tool. A second agent joining mid-session is out of scope.
- **Automatic spec generation.** The tool assembles a Q/A transcript; it does not write a full engineering spec. The synthesis is a ladder, not a destination.
- **Real-time collaboration.** The tool does not stream questions to an external UI. It's a turn-by-turn tool call.
- **Session merging or forking.** One topic, one session, one synthesis. Merging two sessions on the same topic is future work.
