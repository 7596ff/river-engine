# Channel Message Design

*Written: 2026-05-03. Based on architecture trace of river-engine and design conversation.*
*Reviewed by Gemini CLI. Approved with recommendations, incorporated below.*

---

## Problem

The current `handle_incoming` route in `river-gateway` writes messages to a custom-formatted inbox file but never pushes to the agent's `MessageQueue`. The agent polls an empty queue and never wakes. The wire between HTTP handler and agent task is broken.

Beyond the missing wire, the current inbox format is overcomplicated — custom text formatting, read markers baked into the line format, Discord-specific path hierarchy (`conversations/discord/{guild}/{channel}.txt`). The system needs a simpler, adapter-agnostic message layer.

---

## Design

### Channel Logs

One JSONL file per channel. Flat path, no nesting.

**Location:** `channels/{adapter}_{channel_id}.jsonl`

The adapter provides the channel identifier. The gateway sanitizes it into a filename. No guild hierarchy, no adapter-specific directory structure �� that's the adapter's concept, not the engine's.

**Robustness:** The JSONL reader must skip malformed lines. A partial write from a crash or power loss can produce an invalid JSON line. The reader logs a warning and continues to the next line. It does not panic or abort.

### Entry Format

Every entry is a single JSON line with a snowflake ID. The snowflake provides unique identification and temporal ordering in one value.

**Incoming message (from adapter):**

```json
{"id": "snowflake", "role": "other", "author": "cassie", "author_id": "12345", "content": "hello", "adapter": "discord", "msg_id": "123456"}
```

**Agent message (outbound, logged after send):**

```json
{"id": "snowflake", "role": "agent", "content": "good morning cass", "adapter": "discord", "msg_id": "789012"}
```

**Cursor (agent read without speaking):**

```json
{"id": "snowflake", "role": "agent", "cursor": true}
```

### Fields

| Field | Required | Description |
|-------|----------|-------------|
| `id` | yes | Snowflake ID. Unique, sortable, encodes timestamp. Generated by the gateway. |
| `role` | yes | `"other"` or `"agent"`. The agent doesn't care if the speaker is a user, a bot, another agent, or a webhook. It's either me or not-me. |
| `author` | for `other` | Display name of the speaker. |
| `author_id` | for `other` | Adapter-specific unique ID of the speaker. |
| `content` | for messages | The message text. Absent on cursor entries. |
| `adapter` | for messages | Which adapter the message came through or was sent through. |
| `msg_id` | for messages | Adapter-specific message ID. Used for replying to a specific message through the adapter. This is distinct from `id` — the snowflake is the engine's ID for ordering; the `msg_id` is the adapter's ID for interaction (reply, edit, delete). Both must be stored. |
| `cursor` | for cursors | `true`. Marks this entry as a read cursor, not a message. |

### Inbound Flow

```
Discord/adapter sends message
        │
        ▼
POST /incoming on gateway
        │
        ▼
handle_incoming()
  1. Validate auth
  2. Generate snowflake ID
  3. Build JSONL entry: {id, role: "other", author, author_id, content, adapter, msg_id}
  4. Append entry to channels/{adapter}_{channel_id}.jsonl
  5. ON SUCCESS of step 4: push notification to MessageQueue: {channel, snowflake_id}
     ON FAILURE of step 4: return 500, do NOT push notification
  6. Return {"status": "delivered"}
        │
        ▼
Agent wakes (MessageQueue not empty)
        │
        ▼
turn_cycle()
  1. Drain notifications from MessageQueue
  2. For each channel with notifications:
     a. Scan channel log backward for last role:agent entry (message or cursor)
     b. Read forward from that point — everything after is new
     c. If no role:agent entry exists, read the last 50 entries
        from the channel log (or the full log if fewer than 50)
  3. Process messages into context
  4. Run think/act loop
  5. When agent speaks: append role:agent entry to channel log,
     send via adapter
  6. If agent read but didn't speak: append cursor entry
```

### Outbound Flow

```
Agent calls send_message tool during think/act loop
        │
        ▼
send_to_adapter()
  1. POST to adapter's outbound URL
  2. Adapter returns {success, message_id}
        │
        ▼
  3. Generate snowflake ID
  4. Append role:agent entry to channel log:
     {id, role: "agent", content, adapter, msg_id}
  5. Return tool result to agent
```

The agent's outbound message in the channel log serves as an implicit cursor. No separate cursor entry needed when the agent speaks.

### Cursor Rules

The agent's read position in a channel is determined by the last `role: "agent"` entry — whether it's a message or a cursor.

- **Agent speaks** → the message entry is the cursor. Implicit.
- **Agent reads but doesn't speak** → writes an explicit cursor entry. "I read up to here."
- **Agent wakes in a busy public channel** → finds last cursor, reads only what's new since then. Can choose to read more context if needed.
- **Agent has never been in the channel** → no `role: agent` entry exists. Read the last 50 entries from the channel log, or the full log if fewer than 50.
- **Agent is in a channel but hasn't been active for days** → cursor is far back. Agent reads forward from cursor. If the volume is too large, the agent can skip to recent messages and note the gap.

### Notification Queue

The `MessageQueue` carries lightweight notifications, not message content.

Each notification contains:
- `channel`: the channel identifier (e.g., `"discord_general"`)
- `snowflake_id`: the ID of the new message

The notification tells the agent *where* to look, not *what* was said. The channel log is the source of truth. The queue is a wake signal.

Multiple notifications for the same channel between agent turns are fine — the agent reads from its cursor regardless of how many notifications arrived.

A notification is only pushed after a successful log write. If the write fails, no notification is sent. The agent never wakes to find missing data.

## What This Replaces

### Current code to remove:

- `inbox/` module (`inbox.rs` or `inbox/mod.rs`) — `format_inbox_line`, `build_discord_path`, `append_line`, `sanitize_name`
- `conversations/` directory structure and `ConversationWriter` task
- Custom inbox path building in `handle_incoming`
- The `conversations/` migration logic in `server.rs::run()`

### Current code to modify:

- `handle_incoming()` in `api/routes.rs` — replace inbox file write with JSONL append + queue push
- `AgentTask::turn_cycle()` in `agent/task.rs` — replace message queue drain with channel log reading
- `SendMessageTool` and `SpeakTool` — append `role: agent` entries to channel log after sending
- `MessageQueue` in `queue.rs` — change payload from `IncomingMessage` to notification `{channel, snowflake_id}`
- `AppState` — remove inbox/conversation dependencies, ensure snowflake generator is available to the handler

### New code:

- `channels/` module — JSONL read/write, cursor scanning, channel log management, malformed line handling
- Notification struct for the queue
- Channel log reading logic in the agent task

---

## What This Does Not Change

- The adapter protocol — adapters still POST to `/incoming` and receive POSTs on their outbound URL. The adapter doesn't know or care about JSONL.
- The event bus — agent and bystander still communicate through `CoordinatorEvent`.
- The context assembly — how messages get arranged in the context window is a separate concern. This design delivers messages to the agent. Context assembly decides how to present them.
- The memory system — channel logs are not part of the loom or the atomic web. They are communication infrastructure.

## Open Questions

**Channel cleanup.** Channel logs grow indefinitely. When (if ever) do old entries get archived or truncated? The JSONL format makes it easy to rotate — move the old file, start a new one, keep the cursor valid. But the policy isn't specified.

**Multi-channel turns.** If the agent gets notifications from 3 channels simultaneously, does it process all 3 in one turn? Or one channel per turn? The current architecture has a single "active channel" concept. Multiple simultaneous channels may need the agent to prioritize or round-robin.

**Snowflake generation in the handler.** The handler needs access to the `SnowflakeGenerator` to create IDs for incoming messages. `AppState` already holds one via `snowflake_gen`. Just needs to be used in `handle_incoming`.
