# Spectator Redesign Plan Review Prompt

Paste the plan below into Gemini or another reviewer, followed by this prompt. Also paste the spec (`docs/superpowers/specs/2026-05-12-spectator-redesign.md`) for reference.

---

You are reviewing an **implementation plan** for a spectator redesign in a multi-agent orchestration system called river-engine. The plan implements a spec that has already been reviewed twice. Your job is not to re-review the spec — it is to verify that the plan faithfully implements it and that the code will actually compile and work.

This is a critical review. Your job is to **lay bare contradictions** between the plan and the spec, find code that won't compile, catch type mismatches, and identify gaps where the plan promises something but doesn't deliver the code.

## 1. Spec Fidelity

Walk through each section of the spec and verify the plan has a task that implements it:
- Entry formatting with tiered detail — is it exactly as specified? (full text for messages, name-only for tool calls, name + byte count for tool results, heartbeats/cursors filtered)
- Token budget of 16384 — is it applied correctly? Oldest-first? Does the budget cover only entries or the full prompt?
- One sweep, one move — does the plan actually enforce this? Could any code path produce zero moves or multiple moves per sweep?
- Cursor tracking — does `read_cursor` return the right thing? Does the cursor advance only on successful writes?
- Catch-up loop — does it actually loop? What stops it from looping forever?
- Observability — is the system message written to the home channel? Does it use a proper snowflake ID?
- 50-char sanity check — is it in the code?
- Cleanup — is `cleanup_tool_results` called after the move is written?
- Serial processing — is there any async concurrency within the spectator that could cause two sweeps to run simultaneously?

## 2. Compilation Issues

Read the code in each task as if you were the Rust compiler:
- Are all types imported? Do the method signatures match the structs they're called on?
- Does `format_entry` handle all `HomeChannelEntry` variants? Are there match arms for `ToolEntry` with `kind` values other than `"tool_call"` and `"tool_result"`?
- Does `format_entries_budgeted` correctly handle the case where the first entry alone exceeds the budget? (The spec says "always include at least one entry.")
- Does `ChannelLog::read_home_since` exist, or does the plan assume a method that hasn't been created yet? (Check Task 3.)
- Does the spectator's `on_sweep` prompt file get loaded from the right path? The spec says `on-sweep.md`, the old code loaded `on-turn-complete.md`.
- Does `HomeChannelWriter::cleanup_tool_results` take the right arguments? Check the existing signature in `channels/writer.rs`.

## 3. Task Dependencies

- Can Task 1 be implemented and tested independently? Does it depend on types from Task 2 or 3?
- Can Task 4 be implemented without Task 5 (server wiring)? If so, can it be tested?
- Does Task 6 (update `load_moves`) actually change anything? The plan says the current implementation "already reads moves.jsonl." If it already works, is this task necessary?

## 4. Test Coverage

- Does Task 4 (the main rewrite) have any tests? The plan provides tests for Tasks 1, 2, and 3 but not for the sweep logic itself. Is this intentional?
- The entry formatter tests cover individual entries but not edge cases: what about a `MessageEntry` with role `"other"` (the old format)? What about a `ToolEntry` with `kind` neither `"tool_call"` nor `"tool_result"`?
- The moves tests don't test concurrent reads/writes. Is this a concern?

## 5. Data Flow Gaps

- The spectator reads the home channel via `ChannelLog::read_home_since`. But the home channel is also being written to by the agent (via `HomeChannelWriter`). The `HomeChannelWriter` is an actor that buffers writes. Could the spectator read the file before buffered writes have been flushed?
- The spectator writes a system message to the home channel after each sweep. On the next sweep, it reads entries since its cursor. The cursor is the `end` of the last move — but the observability message has a *different* snowflake ID (generated by the spectator's own snowflake generator). Will the next sweep include the spectator's own observability message in the entries? If so, the LLM sees `[spectator] move written covering entries...` as a "new entry" to summarize.
- The plan filters out `system` role messages from the entry formatter... wait, does it? Check. If system messages are NOT filtered, the spectator will summarize its own observability messages.

## 6. What's Missing

- **The prompt file.** The plan references `on-sweep.md` but doesn't create it. Is there a task for writing the actual prompt? Without it, the spectator loads `None` for `on_sweep` and never sweeps.
- **Integration test.** No task tests the full flow: write entries to home channel → trigger TurnComplete → spectator sweeps → move appears in moves.jsonl.
- **Server config.** The plan adds `sweep_interval`, `sweep_token_budget`, and `moves_tail` to `SpectatorConfig` but doesn't add corresponding CLI flags or config file fields for them. Are the defaults sufficient?

Be specific. Cite task numbers and code lines when pointing out issues. Focus on things that would cause compilation failures, runtime panics, or silent data corruption.
