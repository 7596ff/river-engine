# Spectator Redesign Implementation Plan Review

**Plan Reviewed:** `docs/superpowers/plans/2026-05-12-spectator-redesign.md`
**Date:** 2026-05-12
**Reviewer:** Gemini CLI

## 1. Spec Fidelity

The plan generally follows the spec but has several specific deviations and gaps:

*   **Entry Formatting:** Mostly correct, but Task 1 Step 3 includes `system` messages. The spec says "System message: [id] system: content". However, the spec *also* adds a spectator system message for observability. This creates a **Spectator Feedback Loop** (see section 5).
*   **One Sweep, One Move:** Enforced in Task 4 Step 4. However, the catch-up loop in `maybe_sweep` (Task 4 Step 4) might trigger multiple sweeps in one TurnComplete event if the backlog is large.
*   **Catch-up Loop Logic:** The `has_more` check (`last_idx < entries.len() - 1`) will return `true` even if the remaining entries are all filtered (heartbeats/cursors). This leads to a "Catch-up sweep" log entry on every sweep where noise exists at the end of the log.
*   **Observability:** Task 4 Step 4 writes the system message. However, the snowflake ID is generated by the spectator. This message will be read by the *next* sweep as a new entry.
*   **Cleanup Call:** Task 4 Step 4 calls `HomeChannelWriter::cleanup_tool_results` as a **static method**, but it was defined as an **instance method** in the Home Channel plan (and Task 11 of this plan). This is a compilation error.

## 2. Compilation Issues

*   **Static vs Instance Method:** As noted above, `HomeChannelWriter::cleanup_tool_results` is called as `HomeChannelWriter::...` in Task 4 but defined with `&self` in the Home Channel plan.
*   **Match Arm Exhaustiveness:** Task 1 Step 3 `format_entry` matches `ToolEntry` kind. If a `ToolEntry` has a kind other than `"tool_call"` or `"tool_result"`, it returns `None`. This is safe but should be noted.
*   **Budgeting Edge Case:** `format_entries_budgeted` correctly handles the "at least one entry" rule by checking `!lines.is_empty()` before breaking on budget.
*   **Prompt Path:** Correctly uses `on-sweep.md`.

## 3. Task Dependencies

*   **Task 4 Mismatch:** Task 4 (Rewrite SpectatorTask) depends on Task 11 (Cleanup) for the `cleanup_tool_results` method, but Task 11 is at the end of the plan. Task 4 will not compile until Task 11 is implemented (or at least the signature is added).
*   **Task 6 Redundancy:** Task 6 is a good refactor but is not strictly necessary for the feature to function if the existing `load_moves` is already compatible with the JSONL format.

## 4. Test Coverage

*   **Sweep Logic Gap:** There are **zero tests** for the main sweep logic (`maybe_sweep`, `sweep`). This is the most complex part of the redesign (LLM integration, catch-up loops, time gating).
*   **Concurrent Tests:** No tests for concurrent reads of `moves.jsonl` while the spectator is writing.

## 5. Data Flow Gaps

*   **The Spectator Feedback Loop:** Task 4 Step 4 writes a system message to the home channel. Task 1 Step 3 formats `system` messages for the LLM. 
    *   **Result:** The spectator will summarize its own "move written" messages in the next sweep. This is narrative pollution.
    *   **Fix:** `format_entry` should filter out system messages from the `"home"` adapter or with the `[spectator]` prefix.
*   **Writer Flushing:** Since `TurnComplete` happens after the agent has "settled" (and thus finished its writes to the `HomeChannelWriter`), the data should be on disk. However, if the writer actor is slow, the spectator might read an incomplete log.

## 6. What's Missing

*   **Prompt File:** The actual content of `spectator/on-sweep.md` is never defined or created in the plan.
*   **Integration Test:** No end-to-end test (Agent writes -> TurnComplete -> Move Appears).
*   **Server Config:** Task 5 wires the config into `server.rs`, but there's no mention of updating the CLI or config file parsing to allow users to change `sweep_interval` or `sweep_token_budget`.
*   **First Sweep Volume:** A fresh agent with a long history will trigger the catch-up loop until it reaches the end. This is fine, but may be a surprise "token burst" on first run.

## Recommendations

1.  **Refactor `cleanup_tool_results`:** Make it a static method or ensure Task 4 uses the writer instance.
2.  **Filter Spectator Messages:** Update `format_entry` to ignore system messages from the `home` adapter.
3.  **Add Sweep Tests:** Task 4 **must** include tests for the `sweep` state machine, possibly using a mock model client.
4.  **Create Prompt File:** Add a task to create a default `spectator/on-sweep.md`.
