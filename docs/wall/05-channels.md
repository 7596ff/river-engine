# 05 — Channels

A channel is a place where conversation happens: a Discord channel, a DM,
the local chat surface. The channel layer is the engine's communication
ground truth — one append-only JSONL log per channel, plus a lightweight
notification queue that wakes the agent.

## The logs

One file per channel, flat namespace, no nesting:

```
{workspace}/channels/{adapter}_{channel_id}.jsonl
```

The adapter supplies the channel identifier; the engine sanitizes it
into a filename. Guild hierarchies, server names, workspace folders —
those are adapter concepts; the engine's namespace is flat.

Every line is one JSON entry. Three kinds:

```json
{"id":"01JXX...","role":"other","author":"cassie","author_id":"123",
 "content":"hello","adapter":"discord","msg_id":"456"}

{"id":"01JXX...","role":"agent","content":"good morning",
 "adapter":"discord","msg_id":"789"}

{"id":"01JXX...","role":"agent","cursor":true}
```

- `id` — a ULID (ch. 10), generated by the engine. Unique and
  lexically time-ordered; the engine's ordering coordinate.
- `msg_id` — the adapter's own message ID, kept for platform
  interaction (reply, edit, react). Dual IDs, two jobs: the ULID
  orders, the msg_id addresses.
- `role` — `agent` or `other`. Binary on purpose: the engine does not
  care whether the other speaker is a human, a bot, or another agent.
  Me, or not-me.
- A `cursor` entry is a bookmark, not a message: "I read to here."
- `attachments` — optional array; absent or empty when the entry
  carries no files. Each element is
  `{filename, path, mime, size, skipped?}`. For inbound, the engine
  downloads each blob to
  `{workspace}/attachments/{entry_ulid}/{filename}` *before* the
  JSONL line is appended, and `path` is workspace-relative. For
  outbound, `path` is the workspace-relative location the agent
  supplied; the engine does not copy the file. A `path: null` entry
  with a `skipped` reason (`too_large`, `download_failed`) records
  that the attachment existed even when the engine could not store
  it — text content is never lost over a broken blob.

## Cursors

The agent's read position in a channel is the last `role: agent` entry
of any kind:

- **Agent speaks** → the logged outbound message is an implicit cursor.
- **Agent reads without speaking** → it writes an explicit cursor entry
  at settle (ch. 01).
- **Agent has never visited** → no agent entry exists; read the last 50
  entries (or the whole log if shorter).
- **Agent returns after a long absence** → read forward from the
  cursor; if the volume is unreasonable, the agent may skip ahead and
  acknowledge the gap — that is a judgment call, not a protocol rule.

## Inbound flow

```
adapter task receives a platform event
  → engine generates ULID, builds the entry
  → APPEND to the channel log            ← must succeed first
  → push {channel, ulid} onto the notification queue
  → agent task wakes (ch. 01)
```

The order is binding: **log write succeeds before the queue is
touched.** If the write fails, no notification — the agent never wakes
to find missing data. The queue carries pointers, never content; the
log is the single source of what was said. Many notifications for one
channel between turns collapse naturally — the agent reads from its
cursor regardless of how many pings queued.

## Outbound flow

When the agent speaks (ch. 07, the `speak` tool):

```
outbound request → adapter task → platform delivers, returns msg_id
  → engine appends the role:agent entry (with msg_id) to the log
```

The entry is appended only after the platform accepts the message, so
the log records what was actually said, and the entry doubles as the
cursor.

## Robustness

Readers skip malformed lines with a logged warning and keep going — a
crash mid-append may leave a torn line, and a torn line must never
poison a channel. Logs grow without rotation in this design; they are
plain JSONL, trivially archivable by hand if one ever grows obnoxious.

## Contracts

- **Write-then-notify.** A notification is pushed only after its
  entry is durably appended. Failed write → no notification → the
  inbound event is reported as failed to the adapter.
- **Pointers, not payloads.** Queue items are `{channel, ulid}`.
- **Dual IDs.** Every message entry carries the engine ULID and, when
  the platform provides one, the adapter msg_id.
- **Binary roles.** `agent` | `other`. Nothing else, ever.
- **Cursor semantics.** Read position = last agent entry; speak is an
  implicit cursor; read-without-speak writes an explicit one;
  never-visited reads the last 50.
- **Torn-line tolerance.** Malformed JSONL lines are skipped with a
  warning, never fatal.
- **Outbound logged post-acceptance.** The agent entry is appended
  after the platform confirms delivery.
- **Attachments write-then-append.** Every inbound attachment with a
  non-null `path` is durably on disk before the JSONL entry is
  appended. Failed downloads and oversized files append with
  `path: null` and a `skipped` reason; the entry itself is never
  dropped over a broken blob.
